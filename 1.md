# /v1/responses → /v1/messages 可转换性（真实调研版）

说明：本文的 `/v1/messages` 指 Anthropic Claude Messages API（POST /v1/messages）。

## 结论
- “原生完美转换”只能在很小的交集子集成立；大多数真实请求会出现信息丢失或语义变化。
- 主要不一致来源：角色体系不同、输出结构不同（Responses 是 output items 数组）、工具与内置能力类型不同、以及 streaming 事件模型不同。

## 官方事实（两端协议的关键点）

### OpenAI /v1/responses
- 输入消息角色允许 `user`/`assistant`/`system`/`developer`，并可带 `phase`。
- 输入内容类型支持 `input_text`、`input_image`、`input_file`。
- 输出是 `output` items 数组（不是单条 message），可能包含多种 item 类型，例如：
  `message`、`function_call`、`file_search_call`、`web_search_call`、
  `code_interpreter_call`、`image_generation_call`、`computer_call`、
  `mcp_call`、`custom_tool_call`、`reasoning`、`compaction`。
- 结构化输出通过 `text.format`（如 `json_schema` / `json_object`）。
- Streaming 事件示例包含 `response.created`、`response.output_text.delta`、`response.output_text.done`、`response.content_part.done`、`response.completed`。

### Anthropic /v1/messages
- `messages` 仅包含 `user`/`assistant` 角色；系统提示通过顶层 `system` 字段传递，Messages API 没有 `system` 角色。
- 连续的 `user`/`assistant` 会被合并为一个 turn。
- `content` 支持字符串或 content blocks；图片 block 支持 `base64` 或 `url` source。
- 工具调用通过 `tool_use` 和 `tool_result` content blocks；工具定义使用 `input_schema`。
- `tool_result` 必须紧跟对应 `tool_use`，并且在 user 消息的 content 数组中必须排在最前。
- Streaming 事件序列：`message_start` → `content_block_start/delta/stop` → `message_delta` → `message_stop`。
- `usage` 的 token 统计与可见文本不一一对应。

## 可以做到无损转换的严格子集
只有在下列条件**全部满足**时，才可能“原生等价”：
- 仅使用 `user`/`assistant` 角色（不含 `system`/`developer`/`phase`）。
- 输入内容仅为 `input_text` 或 `input_image`，不使用 `input_file`。
- 输出 `output` 仅包含 `type:"message"`，且内容仅为 `output_text`。
- 不使用 Responses 的结构化输出约束（或目标端实现了完全等价的约束）。

最小映射（在该子集内可视为等价）：
- `input_text` → `text`
- `input_image` → `image`（需转换为 Claude 的 `base64` 或 `url` source）
- `output_text` → `text`

## 无法原生等价的部分（只能降级/重写）
- `system`/`developer` 角色与 `phase`：Claude Messages API 无该角色，只能挤到顶层 `system` 或合并为 user 指令，语义会变化。
- 非 `message` 输出项：`function_call`、`file_search_call`、`web_search_call`、`code_interpreter_call`、`image_generation_call`、`computer_call`、`mcp_call`、`custom_tool_call`、`reasoning`、`compaction` 在 Claude 中没有等价的“输出 item”。只能改写为 `tool_use/tool_result`、文本说明或外部状态。
- `input_file`：OpenAI 允许直接把文件作为输入 item；Claude 需要通过内容块或工具结果间接表达，模型看到的上下文不等价。
- Streaming：事件模型完全不同，无法“透传”，只能重组事件序列并维护状态。
- `usage`：Claude 的 token 统计不与可见文本一一对应，无法与 OpenAI usage 做严格等价映射。

## 实务建议
- 若目标是“尽量原生”，必须明确支持子集，并对不支持项执行可预期的降级策略（例如：文本注入、元数据旁路、丢弃并返回警告）。
- 若目标是“严格无损”，只能限制输入输出在上述交集子集内。
